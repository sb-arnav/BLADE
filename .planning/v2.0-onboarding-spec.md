# v2.0 Onboarding Spec — First 60 Seconds + The Hunt

**Status:** working draft, 2026-05-13. Locks the *mechanism* and *first-message language*. Open design decisions are marked `<<DECISION>>`.

**Authority chain:** VISION.md (locked 2026-05-10) → `.planning/decisions.md` 2026-05-13 entries ("first 60 seconds" + "v2.0 onboarding mechanism: agentic hunt"). If this doc contradicts VISION, this doc is wrong.

**Why this exists:** VISION line 281 explicitly says setup-as-conversation needs a design conversation before code. This is that conversation, captured.

---

## The flow in one paragraph

User runs `curl | sh`. Tauri binary downloads, installs, launches. Chat window opens — no wizard, no splash, no restart. BLADE's 2-second pre-scan finds installed agents + API keys + OS context. Message #1 names what it found, says BLADE defaults to a specific model and why, and offers an override. Once a key is verified, BLADE goes hunting: LLM-driven probes of Claude Code conversations, Cursor workspaces, git config, shell history, browser bookmarks — every step narrated live in chat. Result: a synthesized "this is who I think you are" summary, shown back for confirmation, written to `~/.blade/who-you-are.md` (user-editable). Onboarding closes with BLADE asking *"give me one thing you've been putting off — I'll handle it now"* and actually doing it.

Total time: ~5 min if hunt finds rich signals; ~3 min if fresh machine (hunt resolves to a single sharp question + answer-driven probes).

## Act 1 — Install + Pre-scan (≤ 30 sec)

`curl -sSL slayerblade.site/install | sh` downloads ~50MB Tauri binary, installs, auto-launches. **No wizard. No restart. No splash. Chat window opens immediately.**

While the window paints, BLADE runs a parallel 2-second pre-scan. NOT v1.5 Deep Scan reborn — no DB writes, no recursive walks. Just a flat capability inventory used to seed message #1:

| Probe | Reads |
|---|---|
| Agent presence | `which claude`, `which cursor`, `which ollama`, `which gh`, `which aider` |
| API keys (env) | `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, `GROQ_API_KEY`, `GEMINI_API_KEY`, `XAI_API_KEY` |
| API keys (config) | `~/.claude/config`, `~/.cursor/config`, OS keychain entries for known service names |
| Local LLM | TCP probe `:11434` for running Ollama |
| OS + arch | `uname` / `sw_vers` / Windows version detection |
| Default browser | OS-native API (LaunchServices on macOS, registry on Windows, xdg-mime on Linux) |
| Mic | permission check only — no recording |

Lands in an in-memory `InitialContext` struct. Not persisted unless user opts in later.

## Act 2 — Message #1: Key disclosure + override (1 bubble)

```
> Found these on your machine: Claude Code, Cursor, Anthropic
> + Groq keys, Ollama running. 47 tools loaded.
>
> I'll default to Anthropic claude-opus-4-7 for thinking + Groq
> llama-3.3-70b for fast cheap stuff. Routing code work to
> Claude Code when it makes sense.
>
> Use what I found, or paste a different key? Say "use Ollama
> only" to stay 100% local. Say "skip" if you want to start
> talking now and set this up later. Otherwise, just answer
> the next message.
```

Four sentences. Disclosure of what was found. Default and *why* (which model for which job). Override paths including "fully local" and "skip." No "Hi I'm BLADE." No "Welcome."

**Skip semantics:** if user says "skip," BLADE proceeds with auto-detected defaults and captures core command latently from the first task. No durable identity write until BLADE has enough signal.

## Act 3 — The Hunt (LLM-driven, narrated live)

After key verify, BLADE doesn't ask. It goes:

```
> Key verified (Anthropic, claude-opus-4-7). Going to learn
> who you are before I ask anything. Stop me with "stop" if
> you want.
```

Then the LLM-driven hunt begins. **Mechanism, not a hardcoded scanner:**

The hunt is a single LLM session with:
- System prompt: *"You're BLADE, learning who this user is on first launch. You have shell + file-read access via tool calls. Decide what to look at, in what order, to build the user's identity. Sample, don't exhaust. Weight recency aggressively. Narrate every probe to the user in chat. Cap your work at ~50K input tokens — track and surface cost. Surface contradictions as sharp questions rather than asking generic ones."*
- Context: `InitialContext` from pre-scan + `platform_paths.md` (per-OS install conventions, below).
- Tools: `read_file`, `list_dir`, `run_shell` (sandboxed to readonly + no-network), `emit_chat_line`.

Live narration looks like:

```
> Found ~/.claude/projects — reading your 3 most recent
> Claude Code conversations.
> ⤷ Project: clarify-saas. Last touched 2 hours ago.
> ⤷ Building a B2B SaaS for design agencies. Solo founder.
> ⤷ Tech: Next.js + Supabase + Stripe.
>
> Reading git config + recent commits in ~/code.
> ⤷ Commits as "Arnav Maurya <abhinav@pollpe.in>". Mostly
>   TypeScript. ~30 commits this week to clarify-saas.
>
> Cursor workspace also open on clarify-saas. Matches.
>
> Browser default = Brave. Bookmarks bar: Stripe docs,
> Supabase, Tailwind, three founder Twitter accounts.
```

Hunt closes with a synthesis question:

```
> I think I have it. You're Arnav, solo founder building
> Clarify (B2B SaaS for design agencies, Next.js/Supabase/
> Stripe stack). You ship daily. Direct over polite.
>
> Right?
```

User answers; corrections fold in; durable artifact gets written.

## Act 4 — Platform-aware probing (`platform_paths.md`)

The hunt's LLM doesn't know per-OS install conventions a priori. BLADE ships a `platform_paths.md` file loaded into the hunt's context. Structure:

```markdown
## Windows
- Claude Code: %USERPROFILE%\AppData\Local\Programs\Claude\
- WSL detection: `wsl --list --quiet` → for each distro, `wsl which claude`
- WSL Claude conversations: `wsl ls /home/$USER/.claude/projects`
- Cursor: %APPDATA%\Cursor\User\globalStorage\
- Default browser: HKCU\Software\Classes\http\shell\open\command
- Shell history: PowerShell at %APPDATA%\Microsoft\Windows\PowerShell\PSReadLine\

## macOS
- Claude Code: /Applications/Claude.app, ~/.claude/
- Cursor: /Applications/Cursor.app, ~/Library/Application Support/Cursor/
- Default browser: `defaults read com.apple.LaunchServices/com.apple.launchservices.secure`
- Shell history: ~/.zsh_history (default), ~/.bash_history

## Linux
- Claude Code: /usr/local/bin/claude, ~/.local/bin/claude, ~/.claude/
- Cursor: /opt/cursor, ~/.config/Cursor/
- Default browser: `xdg-mime query default x-scheme-handler/http`
- Shell history: ~/.bash_history, ~/.zsh_history, ~/.local/share/fish/
```

**Why a markdown file the LLM reads, not branching Rust code:** adding a new OS or a new path is a doc edit, not a recompile. Reflects the agent-native principle from `.planning/agent-native-audit.md`.

**The WSL-on-Windows case is the canonical "don't fail this" example:** a Windows user has WSL-Ubuntu with Claude Code installed inside it. Naive `which claude` from PowerShell returns nothing. The hunt must know to `wsl --list --quiet` and then `wsl which claude` per distro. Without this, BLADE looks dumb on the most common dev-on-Windows setup.

## Act 5 — No-data fallback (user replaces the machine as input)

If hunt yields nothing — fresh machine, nothing installed, no shell history, no git config — BLADE doesn't pivot to a wizard. It asks ONE sharp question:

```
> Fresh machine. Nothing to read. Going to ask, then search
> on your answer.
>
> What do you do? Not "your job" — the thing you'd point a
> friend at if they asked.
```

User: *"high schooler. building a discord bot that mods my friend's server."*

```
> Got it. Pulling GitHub if you have one — what's your handle?
```

User gives handle → BLADE reads public repos, pinned project, language stats, bio. Hunt resumes with the user's answer as seed.

**The pattern:** when machine has no signal, the user IS the signal. Hunt the answer instead of the disk.

## Act 6 — Synthesis writes a user-editable artifact

Hunt result lands in `~/.blade/who-you-are.md`. **First-class markdown, user-editable any time.** Structure:

```markdown
# Who you are (BLADE's working model)

**Last updated:** 2026-05-13 by hunt
**You can edit this file. BLADE re-reads it every session.**

## Identity
- Name: Arnav Maurya
- Role: Solo founder
- Email: abhinav@pollpe.in

## What you're building
- Clarify — B2B SaaS for design agencies
- Stack: Next.js, Supabase, Stripe
- Last touched: 2026-05-13 (today)
- Status: actively shipping (~30 commits this week)

## How you work
- Bias: direct over polite (BLADE's chat register matches)
- Hours: active 9am–1am IST based on commit times
- Daily rhythm: deep work morning, shipping afternoon, reviews evening
- Tools you trust: Claude Code (delegate code work here), Cursor
  (your IDE), Brave (default browser)

## What's off-limits
- (filled by later autonomy conversation)

## Notes
- Found 3 unanswered investor emails >7 days old (raised in onboarding)
- Has Ollama running locally — wants local fallback for sensitive work
```

This file is BLADE's CLAUDE.md, but for the human, written BY BLADE FROM real signals, editable by the user. Replaces the brittle `brain_identity` table the v1.6 cuts retired.

## Act 7 — First task closes onboarding (NOT a separate mode)

Onboarding does NOT end with "setup complete, now talk to me." It ends with BLADE acting:

```
> Locked your profile. One thing you've been putting off this
> week — I'll handle it now. Don't tell me what setup you need
> for it — just name the thing.
```

User: *"reply to the investor email I haven't answered in 2 weeks"*

BLADE goes — Gmail OAuth pops in browser (contextual consent, first time only), reads the thread, scans for context, asks ONE clarifying question if needed, drafts in chat, asks send/edit/kill, sends. **The first task IS the closing demo of onboarding.** If the user is going to bounce, this is the moment they decide to stay.

## Multi-agent routing — per-task classification, locked once

BLADE doesn't reinvent Claude Code or Cursor. It dispatches:

| Task class | Route |
|---|---|
| Code (multi-file refactor, debugging, feature impl) | Claude Code in managed pane, BLADE drives + shows output in chat |
| Code (single-file edits, quick fixes) | Direct via native `read_file` + `edit_file` tools |
| Browser-heavy (research, form filling, web automation) | BLADE drives via CDP (v1.0's `browser_native`) |
| Long-running research (10+ minutes) | Swarm sub-agent (`swarm.rs`, v1.0) — isolated context, summary returns |
| Vague creative (drafts, pitch, copy) | BLADE handles directly with its own model |
| Capability gap (no native tool exists) | Forge a tool (v1.3's `tool_forge.rs`) |

**Dispatch is asked ONCE during onboarding, then locked:**

```
> Code work — Claude Code or Cursor? I'll send heavy refactors
> to whichever you pick. Light edits I'll do directly.
```

User picks. BLADE writes preference to `who-you-are.md`. Never asks again unless user says "actually use [other] for code now."

## What's NOT in v2.0 onboarding scope

- The autonomy-tier picker (3-tier static or learned policy) — `<<DECISION F4>>` deferred to v2.0 spec sign-off conversation.
- Voice-first onboarding — text-first per research; voice is toggle from day-1+1, not gated at install.
- Avatar / persona customization — never. (Replika anti-pattern.)
- "Conversational profile interview" — never. (vierisid/jarvis anti-pattern.)
- Post-setup restart — never. The whole point.
- Chat history import (ChatGPT export, Claude export) — held for v2.1, no design yet.
- Daily-rhythm capture (when do you want briefings) — happens latently from the first week of use, not asked upfront.

## Open design decisions (`<<DECISION>>` blocks)

### F4 — RESOLVED: autonomy ceiling = inferred from core command, learns from patterns

**Locked 2026-05-13.** Not a setting. Not a tier picker.

- The core command from the hunt (*"Solo founder, runs the whole company"*) IS the autonomy delegation. BLADE asks itself *"what would a competent person in this role do?"* and acts.
- Asks happen ONLY at action-class boundaries (per F6 below): write-to-disk, send-message, spend-money, run-command, cost-threshold crossed.
- Every ask logs: (action class, BLADE's proposed action, user's decision, user's stated reason). Over a week the log becomes the user's policy. Cold-start defaults derive from the core command's verb tense — *"run my company"* → high autonomy; *"help me with homework"* → low autonomy.
- Override via chat any time: *"never touch banking"*, *"always send routine replies"*. Natural language → policy file update.

The wizard tier picker was the easier-wrong answer. See `decisions.md` 2026-05-13 entry for full mechanism + falsification.

### `<<DECISION F5>>` First task as part of onboarding (vs separate "post-onboarding" mode)

Confirmed inline: first task IS the closing move of onboarding (Act 7 above). Codified.

### `<<DECISION F6>>` Visible-thinking selectivity rules

**Position to lock:** Visible thinking surfaces ONLY at:
- Decision-branch moments where BLADE would otherwise have asked the user
- Action-class boundaries: about to write to disk, send a message, spend money, run a command
- Capability-gap moments where BLADE is about to forge a tool
- Cost thresholds: token spend > $0.50 in a single turn

Routine 200ms decisions (which of three files to read first, which tool to call) stay invisible. **This is the line between JARVIS-feel and react-debugger-feel.**

## Falsification (collectively)

- If v2.0 ships and operator UAT (Arnav + Abhinav) flags the live hunt narration as "verbose / slows me down" rather than "wow," narration density is wrong (mechanism stays).
- If token cost per onboarding routinely exceeds ~$3 (Opus pricing at session-start), selectivity prompt is wrong.
- If 30 days post-launch the hunt fails to extract correct identity on >20% of users (fresh machines + corner cases), the no-data fallback is wrong or `platform_paths.md` is incomplete.
- If users flag the `~/.blade/who-you-are.md` file as creepy / surveillance-feeling rather than "transparent / I can edit it" — the artifact framing is wrong, not the data inside it.
- If <50% of v2.0 users complete the first task in Act 7 (drop-off between hunt synthesis and acting), onboarding length is wrong (cut something).
