# Requirements: BLADE

**Defined:** 2026-05-03 baseline; v2.2 milestone scoped 2026-05-14.
**Core Value:** BLADE works out of the box, you can always see what it's doing, and it thinks before it acts.

## v2.2 Requirements — VISION-Close + Goose-Integrate + Launch-Ready

Bring BLADE from "substrate-complete" (v2.1 close) to "VISION-reached at the wedge." Three workstreams in one milestone:

1. **Close the load-bearing VISION gaps** the v2.0/v2.1 audits surfaced — presence is invisible in chat (the fifth primitive has no user-facing narrative); setup-as-conversation doesn't produce a TELOS-shaped artifact; agent-to-agent intro never landed.
2. **Stop building what already exists.** Integrate Goose (Apache 2, pure Rust, Linux Foundation backed) for the provider trait + canonical models registry + SQLite session schema instead of continuing to maintain bespoke equivalents. Adopt OpenClaw's skill-as-markdown directory pattern for the eventual marketplace. Cut bespoke memory complexity (embeddings.rs vector layer) in favor of BM25 + KG, the architecture PAI v5 proved at personal scale.
3. **Prep the forge launch demo + Show HN + Miessler DM.** VISION line 40 names the Twitter-video moment as the wedge. Substrate is ready; assembly is left.

Authority chain: VISION.md (locked 2026-05-10) > V2-AUTONOMOUS-HANDOFF.md > v2.1-MILESTONE-AUDIT.md carry-forward list > REQUIREMENTS.md.

Research substrate: 5 parallel research agents 2026-05-14 (ecosystem scan, Goose internals audit, memory architecture state-of-art, BLADE internal audit vs VISION primitives, launch playbook). Outputs synthesized into the phase scopes below.

Deliberately deferred to v2.3+:
- "Second time destroys it" memory continuity rewrite (cross-session skill persistence + replayable execution memory). Requires schema-level memory rework + sustained operator dogfood signal first. Listed in v2.1 audit as memory fragility (VISION line 136).
- Context-gated privacy questions per VISION §44 (asking privacy questions at decision points, not at setup). Wait for trust-primitive (Cline-style approval gate) + presence narrative to land, then evaluate whether this is still needed.
- Goose's Recipes engine (YAML + minijinja). Real value, but secondary to presence + provider + session foundations.
- Cline-style per-step approval gate. Operator-dogfood feedback decides whether non-developer trust friction warrants this.
- CDN bucket provisioning + Windows ARM64 binaries + shellcheck CI. Release-CI infrastructure, separate concern.
- Held-trio (Body Map / mortality-salience / Ghost Mode) full evaluation — reorganized into a developer pane in this milestone (Phase 59), but the "is it earning weight" verdict still requires real user data.

### Phase 53 — Presence Narrative in Chat (PRESENCE-NARRATE)

The single sharpest VISION-distance move per the v2.2 internal audit. Hormones, vitality, active inference, Evolution Engine are all architecturally complete but produce zero user-facing signal. VISION line 53 explicitly: "memory creates *continuity*; internal state creates *liveliness*. Both matter." Currently only continuity ships. Liveliness ships in this phase.

- [ ] **PRESENCE-EMIT**: Add new chat-line kind `presence` to the existing `BLADE_CHAT_LINE` event stream. Frontend handler in `src/features/chat/useChat.tsx` listens + appends as a distinct system message styled differently from forge/cost lines. Mirror the wiring shape of forge lines (`emit_forge_line` precedent).
- [ ] **PRESENCE-EVOLUTION**: `src-tauri/src/evolution.rs` emits a presence-line when it detects a new MCP server in the catalog, a new capability the user could wire, or an integration opportunity. Format: *"I noticed you use GitHub — want me to wire it in?"* Decision-gated through `decision_gate.rs::evaluate` so it only fires when the source's threshold says yes.
- [ ] **PRESENCE-VITALITY**: `src-tauri/src/homeostasis.rs` emits a presence-line on band transitions (low → med, med → high, high → low). Format: *"Energy's running low — I'll lean on faster models for a bit."* Throttled to at most one transition message per 10 minutes.
- [ ] **PRESENCE-LEARNING**: `src-tauri/src/learning_engine.rs` emits a presence-line when it observes a pattern in user behavior across sessions. Format: *"I'm seeing you sketch React component layouts most mornings — want a shortcut?"* Decision-gated.
- [ ] **PRESENCE-BRAIN-INJECT**: `src-tauri/src/brain.rs` system prompt builder injects the latest 3 presence emissions + current vitality band into the LLM context, so chat responses reflect internal state in tone (not as prose summaries — as a stance modulator).
- [ ] **PRESENCE-TESTS**: 5 unit tests covering each emit site + 1 integration test asserting end-to-end emit → chat render via Tauri runtime mock.

### Phase 54 — Goose Provider Trait Adoption (GOOSE-PROVIDER)

Replace BLADE's bespoke provider abstraction with Goose's Provider/ProviderDef traits + canonical_models.json registry (~1,700 known models). Apache 2 license. Zero architectural risk per the audit — pure Rust trait + data file. Fixes inconsistent multi-provider routing without months of bespoke maintenance.

- [ ] **PROVIDER-TRAIT-PORT**: Copy Goose's `providers/base.rs` trait shape into `src-tauri/src/providers/goose_traits.rs`. Adapt names to BLADE conventions. Keep BLADE's existing auth/keyring layer untouched.
- [ ] **PROVIDER-CANONICAL-MODELS**: Drop Goose's `canonical_models.json` into `src-tauri/data/canonical_models.json`. Add a loader at `src-tauri/src/providers/canonical.rs` that parses + exposes a `lookup(provider, model)` API. Returns context window, pricing per 1M tokens, capabilities.
- [ ] **PROVIDER-MIGRATION**: Migrate `src-tauri/src/providers/{anthropic,openai,groq,gemini,ollama}.rs` to implement the new trait. One PR per provider to keep blast radius bounded. Existing tests must still pass without modification.
- [ ] **PROVIDER-ROUTER-WIRE**: Update `src-tauri/src/router.rs` task classification to consult `canonical_models.json` for context window + pricing when picking the cheapest-sufficient model. Replaces the current hard-coded lookup tables.
- [ ] **PROVIDER-TESTS**: 5 integration tests covering provider parse + model lookup + fallback chain + pricing computation + capability gating. Use mock provider responses, no real API calls at test time.

### Phase 55 — Goose SQLite Session Schema (GOOSE-SESSION)

Adopt Goose's session SQLite schema (session metadata + messages + tool calls as first-class rows) for cross-session continuity + future session-fork + Goose interop. Foundation for "second time destroys it" but doesn't commit to that delivery in v2.2.

- [ ] **SESSION-SCHEMA-PORT**: Define the SQLite schema in `src-tauri/migrations/202605_session_schema.sql`. Tables: `sessions`, `messages`, `tool_calls`, `tool_results`. Match Goose's column shape exactly so future interop is possible.
- [ ] **SESSION-MANAGER**: New module `src-tauri/src/sessions.rs` with `SessionManager`. CRUD operations: `create_session`, `append_message`, `record_tool_call`, `record_tool_result`, `load_session`, `list_sessions`, `fork_session`.
- [ ] **SESSION-COMMANDS-MIGRATE**: `src-tauri/src/commands.rs::send_message_stream` writes to the new session schema instead of (or in parallel with) the current persistence path. One-conversation rollover: existing conversation IDs map to new session IDs.
- [ ] **SESSION-FRONTEND**: `src/lib/tauri/sessions.ts` thin wrapper for `list_sessions` + `load_session` + `fork_session`. UI surface deferred; backend ready.
- [ ] **SESSION-TESTS**: 6 integration tests covering each CRUD op + a fork round-trip + a cross-session message recall.

### Phase 56 — TELOS in Hunt Output (HUNT-TELOS)

Hunt onboarding currently writes `~/.blade/who-you-are.md` as a tool inventory. VISION §42 and PAI's TELOS pattern both say it should also capture Mission / Goals / Beliefs / Challenges so the agent has an *optimization target*, not just a context dump. Closes the setup-as-conversation primitive.

- [ ] **TELOS-PROMPT**: `src-tauri/src/onboarding/hunt.rs` hunt LLM prompt extended to surface 4 follow-up questions (or quietly ingest answers if the conversation already exposed them): Mission (one-line), Goals (3-5 bullets, time-bounded), Beliefs (3-5 things you hold to be true about your work), Challenges (3-5 things in your way).
- [ ] **TELOS-SYNTH**: `src-tauri/src/onboarding/synthesis.rs` writes the synthesized `who-you-are.md` with a YAML frontmatter section (`telos: {mission, goals, beliefs, challenges}`) plus the human-readable body. User-editable.
- [ ] **TELOS-INGEST**: `src-tauri/src/brain.rs` system prompt builder reads `telos:` frontmatter on every chat turn and includes Mission + active Goals in the system prompt. Beliefs + Challenges available on demand.
- [ ] **TELOS-EDIT-FLOW**: Add `blade_open_who_you_are` Tauri command that opens `~/.blade/who-you-are.md` in the user's default editor. Surface a chat command `/edit-self` that triggers it.
- [ ] **TELOS-TESTS**: 4 tests — hunt produces TELOS-shaped output; missing-fields gracefully degrade; user edit round-trip preserves structure; brain prompt includes TELOS when present.

### Phase 57 — Skills as Markdown Directory (SKILLS-MD)

OpenClaw's pattern: skills are plain `.md` files with description + trigger phrases + tool list. AI-installable, crowdsourceable, zero SDK friction. Foundation for the eventual "skills marketplace" VISION roadmap item without building a marketplace yet.

- [ ] **SKILLS-DIR-LAYOUT**: `~/.blade/skills/` directory convention. Each skill is `{name}/SKILL.md` with YAML frontmatter (`name`, `description`, `triggers`, `tools`, `model_hint`).
- [ ] **SKILLS-LOADER**: `src-tauri/src/skills/mod.rs` reads the directory on startup + on filesystem change. Parses YAML frontmatter. Exposes skills via a registry keyed by trigger phrases.
- [ ] **SKILLS-DISPATCH**: `src-tauri/src/commands.rs::send_message_stream` checks skill triggers before normal LLM routing. If a trigger matches with high confidence, the skill's prompt + tool list takes over for that turn.
- [ ] **SKILLS-INSTALL-CMD**: Tauri command `blade_install_skill(url: String)`. Downloads a `SKILL.md` (or a directory tarball) from a trusted URL. Validates schema. Writes to `~/.blade/skills/{name}/`.
- [ ] **SKILLS-SEED**: Ship 5 example skills with BLADE: `summarize-page`, `draft-followup-email`, `extract-todos-from-notes`, `morning-context`, `kill-tabs-i-dont-need`. Each is a SKILL.md, no code.
- [ ] **SKILLS-TESTS**: 5 tests — loader parses valid skill; rejects malformed; trigger matches; install command validates schema; dispatch routes to skill prompt.

### Phase 58 — Embeddings Simplification (MEMORY-SIMPLIFY)

Kill the vector layer in `embeddings.rs`. Keep BM25 + knowledge_graph.rs. The research is conclusive: at BLADE's personal scale (~100k facts, 7 typed categories, 1M-context models), vector retrieval doesn't earn its weight. PAI v5 shipped this. Zep's own paper supports it. Architectural simplification with zero loss of retrieval quality at personal scale.

- [ ] **EMBED-AUDIT**: Document current vector usage. Find all callers of vector retrieval in `embeddings.rs`. Confirm each can be served by BM25 or KG traversal.
- [ ] **EMBED-REMOVE-VECTORS**: Remove the vector index + embedding-generation pipeline. Keep BM25 index. Update `smart_context_recall` to be BM25 + KG only.
- [ ] **EMBED-MIGRATION**: Existing user installations with vector indexes: graceful degrade. On startup, log that vector index is deprecated; BM25 + KG continues without rebuild.
- [ ] **EMBED-TESTS**: Existing retrieval tests must still pass with BM25-only path. Add 3 new tests covering BM25 + KG fusion + typed-category filtering.
- [ ] **EMBED-DEPS**: Remove unused crate dependencies if any (e.g., embedding model loaders) from `Cargo.toml`. Confirm `cargo check` clean after.
