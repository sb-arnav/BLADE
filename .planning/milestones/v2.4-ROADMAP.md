# BLADE — v2.4 Roadmap

**Milestone:** v2.4 — HERMES-PARITY
**Scoped:** 2026-05-17 (post v2.3 close)
**Goal:** Ship a tool-calling harness so faithful to Hermes Function Calling's contract that any Hermes-trained checkpoint plugs in as local default and tool-calling Just Works. Unblocks v2.5+ JARVIS-level capability.

## Phases

```
68 [ ] HERMES-GRAMMAR          (Hermes XML tool-call format emit + parse — plain ChatML XML, NOT special tokens per deep source read)
69 [ ] STREAMING-ACCUMULATOR   (text-stream-live + tool-buffer-until-block-stop, claw-code pattern)
70 [ ] HERMES-PROMPT           (6-field Hermes scaffold + DeepHermes single-concatenated-message for local Hermes)
71 [ ] LOCAL-HERMES            (Ollama + LM Studio first-run Hermes pull as default backend)
72 [ ] CLOSE                   (operator UAT — calendar prompt via Hermes-on-Ollama, NO cloud key, must dispatch tool)
```

                                │

## Success criteria (goal-backward)

Goal: "Operator runs BLADE with Ollama-Hermes as the default backend, no cloud API key configured, sends 'check my calendar' with Gmail MCP installed, and gets real calendar data back via a tool_use dispatch." Falsification:

1. **Hermes XML tool-calls parse round-trip.** Phase 68 fixture tests assert any well-formed Hermes-format tool_call block parses back into the same `ToolCall` struct it was serialized from.
2. **Streaming accumulator never blocks text.** Phase 69 unit test asserts text deltas reach the chat UI within 50ms of SSE arrival, independent of any concurrent tool_use buffering.
3. **Hermes detection doesn't break hosted providers.** Phase 70 test asserts Anthropic + OpenAI provider system prompts are unchanged when the model is not Hermes-family.
4. **Local Hermes accuracy ≥70%** on a small (10-task) tool-calling test set covering calendar / email / file / web operations. If below, the local-default claim is premature; ship without it.
5. **No cloud key required for tool-calling to work end-to-end.** Phase 71 integration test boots BLADE with no API keys, asserts the tool dispatch path completes against a local Hermes-on-Ollama backend.

If 1-5 don't all pass at v2.4-CLOSE, the milestone is not closed — either fix the gap or rescope the "local default" claim.

## Risk register

- **Hermes model size vs. RAM constraint** — Hermes 4 8B is the floor; smaller variants likely don't tool-call reliably. Users with <16GB RAM may struggle. Mitigation: cloud-on-ramp stays first-class; local-Hermes is a path, not forced.
- **Tauri/Ollama integration friction** — Ollama auto-detect already in BLADE, but model-pull UX (5GB download) needs a progress surface. Mitigation: Phase 71 chat-line surface for pull progress (like forge_line pattern).
- **DeepHermes-Atropos BFCL numbers are isolated** — only Berkeley benchmark, no τ-bench / API-Bank cross-check. Real-world accuracy may differ. Mitigation: Phase 72 falsification gate is operator-driven UAT, not benchmark scores.
- **Goose recipe engine + AgentManager unification deferred** — v2.4 stays tactical (Hermes parity); v2.5+ does the architectural collapse (event-sourced conversation, AgentManager keyed by session_id). Risk: v2.4 fixes get partially undone in v2.5 rewrite. Mitigation: keep v2.4 changes narrowly scoped + well-documented so the v2.5 refactor lifts cleanly.

## Out of scope (deferred to v2.5+)

- AgentManager unification (collapse all chat / cron / recipe paths into one AgentSession)
- Event-sourced conversation (immutable append-only event log replacing mutable Vec<ConversationMessage>)
- Cline-style per-step approval gate
- Prompt-cache boundary adoption (v2.5 candidate per the source-read report)
- Atropos-style RL fine-tune of a BLADE-specific tool-calling model (v3.x — long horizon)
