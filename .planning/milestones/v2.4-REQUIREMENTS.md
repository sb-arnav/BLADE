### What "Hermes parity" means concretely

Source read (deep, not abstract — see `.planning/research/v2.4-hermes-source-read.md` by deep-read agent):
- Hermes Function Calling repo (MIT) defines a tool-call grammar with `<tool_call>` and `<tools>` XML-style tokens, a system-prompt template that primes the model to emit tool calls in that grammar, and a Python reference inference loop that parses the stream.
- DeepHermes-ToolCalling-Specialist-Atropos (Apache-2) ships a fine-tuned checkpoint that achieves 2.5–4.6× accuracy improvement over SFT base on parallel + simple tool calls. Two-system-prompt stack: reasoning-first + function-calling-second.
- claw-code's `runtime/src/conversation.rs` is the closest open-source Rust reference for the tool-dispatch loop architecture.

v2.4 BLADE-side deliverables:

1. **Hermes-compatible tool emission.** BLADE's providers/ layer can serialize tools INTO the Hermes XML format (in addition to the Anthropic + OpenAI native formats it already does). Any model that's been trained on Hermes Function Calling can be added as a provider and immediately get reliable tool-calling without bespoke wiring.
2. **Streaming parser gate.** The Phase 62 surgical fix (rip the heuristic) becomes architecturally clean: a dedicated `streaming_gate.rs` module that holds the parser in undecided state until the first non-whitespace event resolves text vs tool. Hermes's `<tool_call>` opener token is the canonical discriminator. Anthropic's `tool_use` block_start is the alternate discriminator.
3. **Two-system-prompt stack** for the local-model path. Reasoning prompt first, function-calling prompt second. Matches DeepHermes's training distribution.
4. **Local model default = Hermes-trained.** When the user runs BLADE without configuring a hosted provider, the default local backend is a Hermes-derived checkpoint (Hermes 4 / DeepHermes / or Hermes-trained Qwen variant — depends on what's permissively licensed at v2.4 ship time). Ollama + LM Studio paths support it.

### Phase 69 — Streaming Tool-Use Accumulator (STREAMING-ACCUMULATOR)

REFRAMED post deep source-read (`.planning/research/v2.4-hermes-source-read.md`): claw-code's pattern is NOT a "parser-switching gate." It's text-stream-live + tool-buffer-until-block-stop. The text path is never blocked; tool input accumulates silently in a buffer; the complete tool_use is emitted only when the SSE content_block_stop frame arrives. This fixes BLADE's v1.1 "40 calls no UI" bug AND enables Hermes parity in one module.

- [ ] **ACCUM-MODULE**: New module `src-tauri/src/providers/streaming_accumulator.rs`. Owns `pending_tool: Option<(id, name, partial_json_buffer: String)>`. State transitions on SSE events: `TextDelta → emit chat_token immediately`, `InputJsonDelta → append to pending_tool buffer`, `ContentBlockStop → drain pending_tool, emit complete ToolUse, dispatch to loop_engine`.
- [ ] **ACCUM-PROVIDER-WIRE**: `providers/anthropic.rs` + `providers/openai.rs` + `providers/groq.rs` + `providers/gemini.rs` SSE parsers feed events through the accumulator. Each provider's SSE shape normalizes to the same internal event type.
- [ ] **ACCUM-HERMES-FORMAT**: For local-Hermes path, the accumulator additionally watches for `<tool_call>...</tool_call>` substring boundaries in the text stream (Hermes models emit them as plain text inside ChatML — no special tokens). On match, divert from text emit to tool buffer.
- [ ] **ACCUM-WIRE**: Phase 62's surgical `if tools.is_empty()` gate becomes architecturally clean: the accumulator handles both paths — empty-tools turn = pure text stream; tool-using turn = text-stream-live + tool-buffer-flush.
- [ ] **ACCUM-TESTS**: 14 unit tests — 4 providers × (text-only / single-tool / parallel-tools) + Hermes XML substring match + malformed-JSON-inside-tool_call + missing-closing-tag.

### Phase 70 — Hermes Prompt Scaffold (HERMES-PROMPT)

REFRAMED post deep source-read: Hermes uses a 6-field system-prompt structure (Role / Objective / Tools / Examples / Schema / Instructions). For local-Hermes models, DeepHermes's two-prompt pattern is a SINGLE CONCATENATED system message (reasoning prompt + function-calling prompt joined), not two separate messages with separate roles. Update prior plan accordingly.

- [ ] **PROMPT-HERMES-SCAFFOLD**: `brain::tools_section_hermes_shape()` helper emits the 6-field structure when `is_hermes_model(provider, model)`. Hosted providers (Anthropic / OpenAI) continue to use native tool definitions — Hermes shape applies to local Hermes-family models only.
- [ ] **PROMPT-CONCAT-LOCAL-HERMES**: When `is_hermes_model` returns true, `build_system_prompt_inner` concatenates the existing BLADE reasoning prompt (identity + telos + memory) with the Hermes function-calling scaffold into a SINGLE system message. Not two messages.
- [ ] **PROMPT-DETECT**: `is_hermes_model(provider: &str, model: &str) -> bool` matches `hermes`, `deephermes`, `nous-hermes`, `hermes-function-calling-specialist`. Provider gate: ollama / lmstudio / any local-model provider; never hosted Anthropic / OpenAI.
- [ ] **PROMPT-TESTS**: 4 unit tests — Hermes detection positive cases, non-Hermes negative cases, hosted-provider path unchanged, concat order preserved (reasoning first).

### Falsification gates

- If Hermes-format serialization lands but no Hermes-trained model actually parses it back → grammar drift, port may have missed a subtle token; revisit against source.
- If the streaming gate adds >200ms p50 latency vs the current naive path → undecided-state buffering is too eager; tune the resolve threshold.
- If two-system-prompt stack makes hosted-provider responses worse (Anthropic / OpenAI) → wrapping the existing prompt builder was wrong; need provider-conditional dispatch.
- If local Hermes-on-Ollama tool-call accuracy is below 70% on a small calendar/email/file test set → the local path isn't competitive yet; ship v2.4 without "local default" claim and queue Atropos-fine-tune for v2.5.
