## v2.3 Requirements — HARNESS-REBUILD-ON-CLAW

Single-focus functionality recovery milestone. v2.2 shipped the presence narrative + Goose foundations + launch artifacts, but operator UAT on the v2.2 build surfaced that **the tool-calling harness silently drops tool_use blocks on conversational-shaped prompts.** "check my calendar" (17 chars, no action-word keyword) hits a fast-path streaming branch at `commands.rs:1822` that never enters `loop_engine::run_loop` — so the LLM emits `tool_use` blocks that get rendered as text. The signature BLADE primitive (forge — VISION §39) cannot fire because the loop it lives in is never reached.

Mac UAT report (2026-05-17) confirmed: `forged_tools` and `forged_tools_invocations` tables stayed at 0 rows across every probe, including the explicit "convert mp4 to GIF" capability-gap test.

This milestone rips the heuristic and routes every chat turn through the tool-dispatching path. Reference architecture: `/_graveyard/2026-05-05/OP_SETUP/rust/crates/runtime/src/conversation.rs:335` (claw-code, Apache-2 clean-room Rust port). Pattern: send-with-tools → parse `pending_tool_uses` → empty=done, non-empty=dispatch-and-loop. No keyword gate.

Authority chain: VISION.md (locked 2026-05-10) > decisions.md 2026-05-17 entry > v2.2 Mac UAT report > REQUIREMENTS.md.

### Phase 62 — Tool Loop Always + Streaming Parser Gate (TOOL-LOOP-ALWAYS)

The headline phase. Two complementary changes:

1. **Rip the heuristic.** The `is_conversational` keyword check at `commands.rs:1822` is removed.
2. **Add a streaming parser gate.** The streaming handler holds in an "undecided" state until the first non-whitespace event in the assistant turn arrives. If that event is a `tool_use` block_start (Anthropic) or a `<tool_call>` opener token (Hermes / OpenAI-style), switch to tool-accumulation mode and feed the loop_engine. Otherwise, continue streaming as text.

This is the Hermes Function Calling pattern (MIT — NousResearch/Hermes-Function-Calling) — structurally identical to parser combinators handling ambiguous grammars. Eliminates the bug AND keeps latency bounded because the gate fires within the first ~100ms of stream data, before any user-visible cost.

See `.planning/research/v2.3-competitor-scan.md` for the full pattern catalogue.

- [ ] **TOOL-LOOP-RIP-FAST-PATH**: Delete the `is_conversational` heuristic (commands.rs:1776–1821) and the fast-path branch that bypasses `loop_engine::run_loop`. Single remaining code path runs through the tool dispatcher.
- [ ] **TOOL-LOOP-STREAM-GATE**: New module `src-tauri/src/providers/streaming_gate.rs`. Wraps the provider SSE parser in a state machine: `Undecided → Text` (on first text token) or `Undecided → ToolAccumulating` (on first `tool_use`/`<tool_call>` event). Once committed, the gate forwards events to either the chat_token emitter or the tool_use accumulator. Attribution comment block to NousResearch/Hermes-Function-Calling (MIT).
- [ ] **TOOL-LOOP-STREAM-DELTAS**: Audit `providers/anthropic.rs`, `providers/openai.rs`, `providers/groq.rs`, `providers/gemini.rs` SSE parsers to confirm they detect tool_use block_start events and accumulate `input_json_delta` through completion. Add unit tests on canned SSE fixtures for each provider.
- [ ] **TOOL-LOOP-NO-TOOLS-GUARD**: Preserve the legitimate optimization — when `tools.is_empty()` (zero tools configured AND no MCP servers connected), skip the streaming gate entirely. This is the original `tools.is_empty()` branch, retained.
- [ ] **TOOL-LOOP-LATENCY-PROBE**: Smoke test measuring p50 latency from `chat_send` to first `chat_token` on a "hi" prompt with full tool set attached. Must stay under 800ms on the local Anthropic path. If it regresses past 1200ms the test fails — the gate must stay streaming-first.
- [ ] **TOOL-LOOP-CALENDAR-FIXTURE**: Integration test asserting "check my calendar" with Gmail MCP installed produces a `tool_use` dispatch event in the trace log, NOT a text-only assistant reply. Falsification test from decisions.md 2026-05-17.
- [ ] **TOOL-LOOP-TESTS**: 8 unit tests covering: (a) Anthropic streaming tool_use parse, (b) OpenAI streaming tool_call parse, (c) Groq streaming tool_call parse, (d) Gemini streaming function_call parse, (e) Hermes `<tool_call>` tag parse for the future local-model path, (f) gate stays in Text mode for pure-text streams, (g) gate switches to ToolAccumulating on first tool_use event, (h) preserve existing `loop_engine::run_loop` happy path.

### Phase 63 — Forge: GitHub-First Search (FORGE-GITHUB-FIRST)

Operator surfaced this in the v2.3 kickoff: "first blade goes and tools for if the tool is available on github right?- it should cause it is easier." Reuse > rewrite. Before forge writes a tool from scratch, query GitHub for an existing MCP server / Tauri command / shell script that matches the capability gap. Only fall back to scratch on miss.

- [ ] **FORGE-GH-SEARCH-CMD**: New module `src-tauri/src/forge/github_search.rs`. Single function `search_existing_tool(capability: &str, gap_context: &CapabilityGap) -> Vec<ToolCandidate>` that queries GitHub's search API for `mcp-server-{capability}`, `{capability}-mcp`, and topic:mcp + topic match. Returns up to 5 candidates ranked by stars + last commit recency.
- [ ] **FORGE-GH-EVAL**: For each candidate, fetch the README + package manifest. LLM-eval: "does this tool plausibly solve the capability gap?" Returns yes/no + 1-line reason. Reuses the existing forge LLM client.
- [ ] **FORGE-GH-INSTALL**: On positive eval + user approval, run the candidate's documented install command in a sandboxed shell. Capture stdout + stderr + exit code. Re-fetch tool list. Emit a `forge_line` of kind `installed_from_github` with the repo URL.
- [ ] **FORGE-GH-FALLBACK**: If GitHub search returns 0 candidates OR all candidates fail eval, fall through to the existing forge-from-scratch path. Existing behavior preserved.
- [ ] **FORGE-GH-LATENCY**: GitHub search must complete in <8s (cold cache) or <500ms (warm). Cache key = SHA256 of capability string. Cache TTL = 24h. If slow, FORGE-GH-FALLBACK fires.
- [ ] **FORGE-GH-TESTS**: 4 integration tests against a mock GitHub API surface (no real GitHub hits during CI). Cover: (a) hit-and-install, (b) hit-but-eval-rejects, (c) miss-and-fallback, (d) timeout-and-fallback.

### Phase 64 — Mac Developer ID Signed Release (MAC-SIGNED-RELEASE)

Mac UAT report B2: every install regenerates the bundle ID, every macOS TCC entitlement (Screen Recording, Mic, Accessibility, Downloads/Documents/Desktop/iCloud) re-prompts from scratch, and every keychain ACL re-grant requires the user's macOS login password. This is the worst single piece of upgrade UX BLADE has. Operator confirmed in v2.3 kickoff: "BLADE asks for permission a lot man — others Apps don't even ones like claude and stuff."

- [ ] **MAC-SIGN-DEV-ID**: Acquire a Developer ID Application certificate (operator-action). Wire `tauri.conf.json` macOS bundle config to use it. Set stable `CFBundleIdentifier = site.slayerblade.blade`.
- [ ] **MAC-SIGN-CI**: GitHub Actions release.yml updated to sign the .app + .dmg with the Developer ID. Notarize via `xcrun notarytool`. Staple the ticket. Reuse the existing `latest.json` updater manifest.
- [ ] **MAC-SIGN-VERIFY**: Post-release verification step: `codesign -dv` reports the real Developer ID, not `flags=0x20002(adhoc,linker-signed)`. Asset SHA256 published in the release notes.
- [ ] **MAC-SIGN-DOCS**: README install section adds the "install + verify codesign" path for security-conscious users.

### Phase 65 — Status Indicator Render (STATUS-INDICATOR-RENDER)

Operator surfaced in v2.3 kickoff: "there is no indication of Blade working after sending a prompt — you give prompt — you wait — you get the answer — there is no indication during the waiting time." `blade_status: "processing"` already emits from the Rust side; the chat UI doesn't render it visibly between user-send and first `chat_token`.

- [ ] **STATUS-RENDER-PROCESSING**: `src/features/chat/useChat.tsx` subscribes to `blade_status` and renders a "Working…" line below the latest user message until the first `chat_token` arrives. Distinct visual style from assistant messages (lighter color, possibly an animated dot).
- [ ] **STATUS-RENDER-COMPACTING**: Same render path for the `compacting` status — surface "Compacting earlier conversation…" so the user knows BLADE isn't hung when a 5+ turn conversation crosses the compaction threshold.
- [ ] **STATUS-RENDER-TOOL**: On `tool_use_start` events from the loop engine, render "Calling {tool_name}…" inline. Replace on `tool_result` with the result preview (existing forge-line pattern).
- [ ] **STATUS-RENDER-TESTS**: 2 Playwright tests asserting the indicator appears within 100ms of `chat_send` and disappears within 100ms of the first `chat_token`.

### Phase 66 — Onboarded Flag From State (ONBOARDED-FLAG-FROM-STATE)

Mac UAT report fresh-onboarding observation: `config.json.onboarded: true` and `persona_onboarding_complete: true` were pre-set on first run, before any TELOS answers were given. The flags should reflect actual state, not be pre-set.

- [ ] **ONBOARDED-DEFAULT-FALSE**: `BladeConfig::default()` and `DiskConfig::default()` both initialize `onboarded = false`. Fresh installs must observe `onboarded: false` until the user completes the hunt.
- [ ] **ONBOARDED-WRITE-ON-COMPLETE**: `onboarding/hunt.rs` writes `onboarded = true` only after the hunt synthesis turn completes successfully and the who-you-are.md file is written.
- [ ] **PERSONA-FLAG-FROM-STATE**: Same pattern for `persona_onboarding_complete`. Writes on synthesis success only.
- [ ] **SKILL-MODEL-HINT-UPGRADE**: Mac report B8 — all 5 seeded SKILL.md files reference `claude-3-5-sonnet-20241022`. Bump to `claude-sonnet-4-20250514` to match the runtime default. Single-line change in `skills_md/seed.rs`.
- [ ] **ONBOARDED-TESTS**: 1 integration test asserting fresh install + zero user input ends with `onboarded: false` in config.json.
