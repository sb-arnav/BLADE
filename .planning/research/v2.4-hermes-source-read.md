## Top corrections to prior abstract

1. **Hermes does NOT use special tokens.** Prior abstract framed `<tool_call>` as a "special token gate." Direct read confirms: it's plain text XML-style tags inside a ChatML template. No tokenizer surgery needed. Plain substring boundary detection works.
2. **Hermes does NOT support text + tool in the same assistant turn.** Prose turns and tool-call turns are disjoint per the reference inference loop. Models trained on Hermes Function Calling emit either prose OR `<tool_call>` blocks per turn, never both.
3. **claw-code's `conversation.rs:335` BUFFERS tool input until block_stop**, doesn't dispatch mid-stream. Text streams to UI immediately; `InputJsonDelta` SSE events accumulate into a per-tool buffer; `ContentBlockStop` flushes the complete `ToolUse` block. The full tool_use isn't dispatched mid-stream.
4. **DeepHermes's two-prompt pattern is a SINGLE concatenated system message**, not two messages with separate roles. Reasoning prompt first, function-calling prompt second, joined.
5. **DeepHermes's reported gains are BFCL-only.** Berkeley Function Calling Leaderboard. No τ-bench / API-Bank / T-bench numbers. 2.5–4.6× parallel/simple gains are real but base Llama-3.1-8B parallel score is 0.10 (very low), so the multiplier overstates absolute capability.

## Pattern lift targets — ranked by load-bearing-ness for BLADE v2.4

### 1. Streaming text + buffered tool_use accumulator (HIGHEST PRIORITY)

**Source:** claw-code `rusty-claude-cli/src/main.rs:4200-4250` (MIT)

The pattern that fixes BLADE's v1.1 "40 calls no UI" bug AND enables Hermes parity simultaneously:
- Owns `pending_tool: Option<(id, name, String)>` accumulator
- On SSE `TextDelta` → write to chat_token emit immediately (live streaming text)
- On SSE `InputJsonDelta { partial_json }` → append to `pending_tool` buffer (silent accumulation)
- On SSE `ContentBlockStop` → drain `pending_tool`, emit `AssistantEvent::ToolUse { id, name, input }`
- Text never blocks tool_use; tool_use never streams to UI

BLADE port target: `src-tauri/src/providers/streaming_gate.rs` (Phase 69). Single ~150 LOC state machine, no buffering of text, only tool_use input gets accumulated.

### 2. Outer agentic loop with hooks (HIGH PRIORITY)

**Source:** claw-code `runtime/src/conversation.rs:286-460` (MIT, ~175 LOC of loop)

State machine to copy:
```
loop {
  events = api_client.stream(request).await        // returns Vec<AssistantEvent>
  message = build_assistant_message(events)         // collapses to single ConversationMessage
  tool_uses = message.filter(ContentBlock::ToolUse) // pending dispatch
  if tool_uses.is_empty() { break }                 // exit on no-tool turn
  for (id, name, input) in tool_uses {
    pre_hook_result = run_pre_tool_use_hook(name, input)?
    if pre_hook_result.denied { push deny ToolResult; continue }
    actual_input = pre_hook_result.updated_input.unwrap_or(input)  // hook can mutate
    enforce = permission_enforcer.check(name, actual_input)?
    if enforce.denied { ... }
    output = tool.execute(actual_input).await
    post_hook_result = run_post_tool_use_hook(name, actual_input, output)?
    push ToolResult { id, content: output } to session
  }
  // outer loop re-enters with new session state
}
```

BLADE port target: refactor `loop_engine::run_loop` to match this shape. Three hook points only (Pre, Post, PostFailure). Hooks can mutate tool input AND override permissions.

### 3. Hermes 6-field prompt scaffold (MEDIUM PRIORITY)

**Source:** Hermes-Function-Calling `prompt_assets/sys_prompt.yml` (MIT, 38 lines)

Structure: Role → Objective → Tools (interpolated JSON schema list) → Examples (few-shot tool-call demos) → Schema (the literal JSON shape model must emit) → Instructions (constraints like "max 10 iterations", "use XML tags").

BLADE port target: `brain::build_system_prompt` factor in a `tools_section_hermes_shape()` helper that emits this structure when the model is Hermes-family. Hosted providers (Anthropic / OpenAI) continue to use native tool definitions.

### 4. Two-prompt concatenation for local-Hermes path (MEDIUM PRIORITY)

**Source:** DeepHermes-ToolCalling-Specialist-Atropos model card (Apache-2)

When model name matches `hermes` / `deephermes` / `nous-hermes`:
- Build reasoning prompt (BLADE identity + telos + memory — existing brain.rs output)
- Append function-calling prompt (the Hermes 6-field structure from above)
- Send as a SINGLE system message, not two separate ones
- Result: model does `<think>...</think>` reasoning then `<tool_call>...</tool_call>` dispatch in one assistant turn (well — across two assistant turns, since prose+tool are disjoint per finding #2)

BLADE port target: `brain::build_system_prompt_inner` adds a `if is_hermes_model(provider, model) { concat reasoning + function-calling }` branch.

### 5. System prompt cache boundary sentinel (LOW PRIORITY, FUTURE)

**Source:** claw-code `runtime/src/prompt.rs:134` (MIT)

Explicit `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` const splits the cacheable prefix (intro, output style, system, doing-tasks, actions) from the per-request tail (environment, project context, config). Anthropic prompt-cache header set against the prefix slice.

BLADE port target: defer to v2.5+ when prompt-cache adoption is on the roadmap. Out of v2.4 scope.

### 6. Read-only bash classifier (BONUS, LOW EFFORT)

**Source:** claw-code `runtime/src/permission_enforcer.rs:155` (MIT, ~50 LOC static allowlist)

Hardcoded list of ~50 safe bash binaries (cat, ls, grep, jq, git, gh, etc.) with negative filters for `>` `>>` `-i` `--in-place`. Drops into BLADE's `bash_validation.rs` (which already exists) as the "safe for autorun" allowlist.

## Confirmed: Hermes does NOT ship a Rust streaming parser

- Hermes Function Calling repo is 100% Python.
- Closest Rust ports exist in `mistral.rs/tools/` and `llama.cpp/server/tool_calls_hermes.cpp`.
- For BLADE, the right move is a ~150-line custom Rust streaming parser modeled on claw-code's pattern (#1 above), not lifting from `mistral.rs`.
