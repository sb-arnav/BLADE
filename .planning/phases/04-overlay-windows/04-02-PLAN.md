<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user keyboard → QuickAsk window | Untrusted user text entering the query input |
| QuickAsk window → Tauri invoke | `quickaskSubmit` passes user query to Rust pipeline |
| localStorage (quickask window scope) | 5-item history persisted in window-local localStorage |
| main → quickask cross-window emit | Rust emits chat_token / chat_done / blade_message_start to this window |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-02-01 | Tampering | localStorage history tampering | accept | Self-tampering only (no cross-origin in Tauri webview). Worst case: user sees a tampered past-query — no secret exposure, no privilege escalation. |
| T-04-02-02 | Information Disclosure | streaming response visible on blur | mitigate | Blur event hides window → response no longer rendered. Subsequent screen capture sees nothing (content_protected only applies to ghost_overlay per D-09; quickask is deliberately visible but auto-hides). |
| T-04-02-03 | DoS | Rapid Cmd+Enter spam triggers multiple streams | mitigate | `busy` flag gates submit; concurrent `quickaskSubmit` calls are rejected frontend-side. Rust backend serializes streaming via existing lock. |
| T-04-02-04 | Tampering | Wake word event spoofing via __TAURI_EMIT__ test hook (DEV only) | accept | DEV-only test shim; production builds don't expose the hook. |
| T-04-02-05 | Denial of Service | Very long query exhausts context window | accept | Rust backend already computes token ratio + emits `blade_token_ratio` (Phase 3). User-facing warning when > 0.65 — but Phase 4 QuickAsk doesn't render this pill (deferred to main chat panel). Accepted risk: oversized quickask query may fail at provider layer; user sees error toast. |
</threat_model>
