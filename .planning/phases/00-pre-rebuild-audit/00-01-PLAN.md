### Classification Rules (D-14 is the locked policy)

- **cross-window** = the event genuinely needs to reach 2+ Tauri windows (main + overlay, main + hud, etc.). Keep as `emit_all`.
- **single-window** = the event is consumed by exactly one window. Convert to `emit_to(label, ...)` — inline the proposed replacement call.
- **ambiguous** = you cannot tell from reading the emit site alone; flag for synthesis review in Wave 2.

Heuristics for classifying:
- Event name contains `overlay`, `hud`, `ghost`, `quickask` → likely single-window (target that window's label).
- Event name prefixed with `blade_` and consumed by chat/dashboard/settings → likely single-window (main).
- Event used for system-wide state (hormone updates, godmode tier changes, perception fusion) → cross-window is plausible; mark ambiguous if unclear.
- If the same event name is emitted from multiple sites that target different windows → ambiguous.

### Scan Method

1. Grep for all three patterns in `src-tauri/src/**/*.rs`:
   - `emit_all(`
   - `app.emit(` (the Tauri 2 AppHandle.emit is also a cross-window broadcast by default)
   - `emit_to(`
2. For each hit, record: file, line, the exact event name literal, the payload type (struct name or "Value"), and classify per the rules above.
